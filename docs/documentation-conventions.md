# Documentation Conventions

## 목적

이 문서는 README와 설계 문서에 어떤 정보를 어떤 형식으로 남길지에 대한 기준을 정의한다.

## README 작성 원칙

- README에는 프로젝트 개요, 아키텍처, 핵심 시퀀스, ERD, 장애 대응, 실행 방법, 현재 구현 범위를 포함한다.
- 구현 완료 기능과 예정 기능을 구분해서 작성한다.
- 기술 이름만 나열하지 말고, 해결하려는 문제와 설계 의도를 함께 설명한다.

## 다이어그램 작성 원칙

- 시퀀스 다이어그램은 `성공`, `부분 실패`, `장애`를 분리해서 작성한다.
- 장애 시나리오는 Kafka 장애, DB 장애, 전체 장애처럼 운영 관점에서 구분한다.
- ERD에는 실제로 사용하는 컬럼과 식별자를 반영한다.
- 코드에서 상태값이나 이벤트명이 바뀌면 다이어그램도 함께 갱신한다.

## Swagger 작성 원칙

- 외부 공개 API에는 `@Tag`, `@Operation`을 작성한다.
- 요청값과 응답값이 비즈니스 흐름에서 어떤 의미를 갖는지 설명한다.
- 웹훅, 결제 승인, 보상 트랜잭션처럼 흐름상 중요한 API는 설명을 생략하지 않는다.

## 용어 사용 원칙

- 문서에서는 코드와 동일한 용어를 사용한다.
- `paymentId`, `paymentKey`, `eventId`처럼 역할이 다른 식별자를 혼용하지 않는다.
- 상태값은 `CREATED`, `PENDING`, `PAID`, `FAILED`처럼 코드 기준으로 표기한다.
