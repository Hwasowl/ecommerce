# Exception Conventions

## 목적

이 문서는 서비스 전반에 공통으로 적용할 예외 처리 규칙을 정리한다.

## 기본 원칙

- 사용자 입력 오류와 비즈니스 오류는 커스텀 예외로 표현한다.
- 전역 예외 핸들러에서 공통 응답 포맷으로 변환한다.
- 응답 포맷은 `code`, `message`, `timestamp`를 기본으로 유지한다.
- 내부 예외 스택을 그대로 클라이언트에 노출하지 않는다.

## 에러 코드 규칙

- 주문 서비스는 `ORDER-xxx`
- 결제 서비스는 `PAYMENT-xxx`
- 재고 서비스는 `INVENTORY-xxx`

## 상태코드 기준

- `400`: 요청값 검증 실패
- `404`: 조회 대상 없음
- `409`: 상태 충돌 또는 중복 처리 충돌
- `422`: 요청은 유효하지만 현재 비즈니스 규칙상 처리 불가
- `500`: 예상하지 못한 내부 오류

## 메시지 작성 규칙

- 메시지는 API 소비자가 이해할 수 있게 명확하게 작성한다.
- 외부 시스템 장애와 내부 시스템 장애는 구분된 메시지와 로그를 남긴다.
- 운영 로그와 클라이언트 응답 메시지가 다를 필요가 있으면 로그를 별도로 남긴다.

## 웹훅과 비동기 처리 예외

- DB 저장이 끝나기 전에는 성공 응답을 반환하지 않는다.
- Kafka만 장애인 경우에는 Outbox 저장 완료 여부를 기준으로 응답을 결정한다.
- DB가 장애인 경우에는 Kafka가 살아 있어도 이벤트를 먼저 발행하지 않는다.
- PG 재시도가 필요한 상황은 예외 처리 정책과 HTTP 응답 코드를 함께 설계한다.
